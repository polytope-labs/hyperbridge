# Consensus Relayers

  <img src="/relayers.png" alt="Hyperbridge relayers" />
  <figcaption style={{ textAlign: "center" }}>Hyperbridge relayers</figcaption>
</figure>

Consensus relaying is one of three responsibilities the [tesseract relayer](/developers/network/relayer) carries out. The other two are [messaging relaying](/developers/explore/relayers/messaging-relayers) and [outbound delivery](/developers/explore/relayers/outbound-relayers). Like the others, it is fully permissionless. There is no staking or whitelisting required to submit consensus updates. The role is simple to describe: observe the state of a source blockchain and submit a verifiable consensus proof to Hyperbridge.

Hyperbridge's security and functionality rely on a continuous, timely flow of consensus proofs from its connected blockchains. To keep that flow alive, Hyperbridge runs an on chain incentive mechanism that economically rewards relayers for their contributions. The mechanism is autonomous, permissionless, and designed to foster a competitive ecosystem of operators.

## Rewarding Consensus Proofs

The incentive mechanism compensates the relayer for keeping Hyperbridge's view of external chains current. This service is fundamental. Without fresh consensus proofs, cross chain messages cannot be securely verified or delivered.

Rewards are paid in Hyperbridge's native currency. On mainnet and testnet that is `$BRIDGE`, although the actual asset is the `Currency` type configured by the runtime, so governance can retarget it without a code change. Alongside the currency reward, the same relayer account is credited a separate reputation asset that accumulates with every successful submission.

## The Reward Mechanism: A Detailed Breakdown

The path from proof submission to reward is deterministic and easy to audit. It is driven by specific on chain events in the Hyperbridge runtime.

1. **The triggering action.** A relayer monitors a connected external chain, for example an Ethereum L2. When that chain finalizes a new block or a set of blocks, the relayer constructs a valid consensus proof and submits it as a transaction to Hyperbridge.
2. **State verification and update.** The Hyperbridge runtime receives the proof and verifies it. If the proof is authentic and advances the consensus state of the source chain (in other words, it represents a newer block height), Hyperbridge accepts it.
3. **The `StateMachineUpdated` event.** On successful verification and state update, the runtime emits a `StateMachineUpdated` event. This is the definitive, auditable record that a state machine has been advanced by the submitting transaction.
4. **Automated reward issuance.** Reward issuance is tied directly to that event. The protocol disburses a fixed amount of the configured currency from the treasury pallet to the account recovered from the proof submitter's signature, and mints reputation to the same account.

## A Competitive, Permissionless Environment

The incentive structure produces a healthy, competitive market. Only the **first relayer** to submit the proof for a new state update receives the reward for that update. This pushes operators to optimize for low latency and efficiency, so Hyperbridge tracks external state changes as quickly as possible.

The model is permissionless by design. No staking or whitelisting is required to participate. Any operator can run a tesseract instance, start competing to submit proofs, and earn rewards. That openness is what gives the network a decentralized foundation.

## Distinction from Other Relayer Responsibilities

It is worth distinguishing these automated consensus rewards from the fees relayers earn for relaying individual messages, and from the costs of outbound delivery.

* **Consensus rewards (`$BRIDGE` plus reputation).** Protocol level rewards paid by Hyperbridge for the public good of maintaining network wide state liveness. Funded from the Hyperbridge treasury pallet.
* **Message delivery fees (user paid).** Application specific fees paid by users on the source chain to incentivize the messaging relaying role to deliver their request or response. See [Messaging Relayers](/developers/explore/relayers/messaging-relayers).
* **Outbound delivery (incentivized via the BEEFY proof reward, with direct relayer incentivization).** The fan out of accepted Hyperbridge proofs back to every configured destination chain. See [Outbound Relayers](/developers/explore/relayers/outbound-relayers).

A single tesseract process performs all three roles. The economic incentives are distinct, and they serve different purposes within the Hyperbridge ecosystem.
