# Placing Orders

Intent orders let users express what they want â€” "swap 1000 USDC on Ethereum for 990 USDC on Arbitrum" â€” and leave the execution to a competitive market of solvers. Users escrow input tokens upfront; solvers compete to fill the output; the best bid wins.

The `IntentGatewayV2` contract handles escrow, fee accounting, and cross-chain settlement. The `@hyperbridge/sdk` `IntentGateway` class wraps the full lifecycle â€” from encoding and broadcasting the `placeOrder` transaction to polling for solver bids, submitting the winning `UserOperation`, and streaming real-time status updates â€” into a single async generator.

## Setup

The setup differs slightly depending on whether you are placing a same-chain or cross-chain order.

### Bundler Services

A bundler URL is required for the destination chain. Any ERC-4337-compatible bundler works. [Alchemy](https://alchemy.com/account-abstraction) provides bundler support as part of its RPC endpoints (with Account Abstraction enabled):

| Provider | URL format |
|----------|-----------|
| [Alchemy](https://alchemy.com/account-abstraction) | `https://{chain}.g.alchemy.com/v2/YOUR_KEY` |

### Cross-Chain Orders

```typescript lineNumbers title="setup-cross-chain.ts" icon="typescript"
import { EvmChain, IntentsCoprocessor, IntentGateway } from "@hyperbridge/sdk"
import { createWalletClient, http } from "viem"
import { privateKeyToAccount } from "viem/accounts"

// Source chain â€” where the user places the order and tokens are escrowed
const sourceChain = await EvmChain.create("YOUR_SOURCE_RPC_URL")

// Destination chain â€” bundlerUrl is REQUIRED; solvers submit UserOperations via this bundler
const destChain = await EvmChain.create(
  "https://arb-mainnet.g.alchemy.com/v2/YOUR_ALCHEMY_KEY",
  "https://arb-mainnet.g.alchemy.com/v2/YOUR_ALCHEMY_KEY",
)

// Hyperbridge coprocessor â€” hosts the order auction and manages solver bids
const coprocessor = await IntentsCoprocessor.connect("wss://nexus.rpc.polytope.technology")

const intentGateway = await IntentGateway.create(sourceChain, destChain, coprocessor)

const account = privateKeyToAccount("0xYOUR_PRIVATE_KEY")
const walletClient = createWalletClient({
  account,
  transport: http("YOUR_SOURCE_RPC_URL"),
})
```

<Callout type="warn">
The `bundlerUrl` on the destination chain is **required** for cross-chain orders. `IntentGateway` uses it to submit the solver's `UserOperation` to the ERC-4337 bundler on the destination chain. Without it, bid submission will fail at the `BID_SELECTED` stage.
</Callout>

### Same-Chain Orders

For same-chain swaps, source and destination are the same chain. The `bundlerUrl` is still required â€” the solver fills the order on the same chain via an ERC-4337 `UserOperation`.

```typescript lineNumbers title="setup-same-chain.ts" icon="typescript"
import { EvmChain, IntentsCoprocessor, IntentGateway } from "@hyperbridge/sdk"
import { createWalletClient, http } from "viem"
import { privateKeyToAccount } from "viem/accounts"

const RPC_URL     = "https://base-mainnet.g.alchemy.com/v2/YOUR_ALCHEMY_KEY"
const BUNDLER_URL = "https://base-mainnet.g.alchemy.com/v2/YOUR_ALCHEMY_KEY"

// Source and destination are the same chain for same-chain swaps
const chain = await EvmChain.create(RPC_URL, BUNDLER_URL)

// Hyperbridge coprocessor â€” hosts the auction even for same-chain orders
const coprocessor = await IntentsCoprocessor.connect("wss://nexus.rpc.polytope.technology")

// Pass the same chain instance for both source and dest
const intentGateway = await IntentGateway.create(chain, chain, coprocessor)

const account = privateKeyToAccount("0xYOUR_PRIVATE_KEY")
const walletClient = createWalletClient({
  account,
  transport: http(RPC_URL),
})
```

<Callout type="info">
The ERC-4337 bundler URL is attached to the `EvmChain` instance â€” via the second argument to `EvmChain.create()` or the `bundlerUrl` field in `EvmChainParams`. `IntentGateway` reads it automatically from `dest.bundlerUrl`.
</Callout>

## How `placeOrder` Works

Understanding what happens on-chain helps you construct orders correctly.

```solidity
function placeOrder(Order calldata order, bytes32 graffiti) external payable;
```

### Data structures

```solidity
struct Order {
    bytes32 user;           // Set by placeOrder to msg.sender
    bytes source;           // Set by placeOrder to current chain ID
    bytes destination;      // Target chain state machine ID
    uint256 deadline;       // Block number deadline for fills
    uint256 nonce;          // Set by placeOrder (auto-incrementing)
    uint256 fees;           // Dispatch fees in fee token
    address session;        // Session key for solver selection (address(0) to disable)
    DispatchInfo predispatch; // Pre-order execution (swap before escrow)
    TokenInfo[] inputs;     // Tokens escrowed from user
    PaymentInfo output;     // What the solver must provide
}

struct TokenInfo {
    bytes32 token;   // Token address as bytes32; bytes32(0) = native token
    uint256 amount;
}

struct PaymentInfo {
    bytes32 beneficiary;  // Recipient of output tokens
    TokenInfo[] assets;   // Output tokens solver must provide
    bytes call;           // Optional calldata executed after fill (via CallDispatcher)
}

struct DispatchInfo {
    TokenInfo[] assets;   // Assets sent to CallDispatcher before order
    bytes call;           // Calldata for pre-order execution
}

```

### Protocol fees

Before escrowing, the contract deducts a protocol fee from each input amount:

```
protocolFee = input.amount Ã— protocolFeeBps / 10_000
reducedAmount = input.amount âˆ’ protocolFee
```

The fee is retained in the gateway as dust (emitting `DustCollected`). Per-destination fee overrides (`_destinationProtocolFees`) take precedence over the global `_params.protocolFeeBps` when set.

**The commitment hash is computed over the fee-reduced inputs.** Solvers read `reducedAmount` from the `OrderPlaced` event and only need to match those amounts.

### Dispatch fees

`order.fees` is a tip paid to the solver who fills the order. For same-chain orders it compensates the filler for gas; for cross-chain orders it covers the Hyperbridge relayer fees required to deliver the `RedeemEscrow` message back to the source chain. Setting it too low makes the order unattractive to solvers â€” leave it as `0n` and the SDK will estimate an appropriate amount automatically.

The contract accepts the fee in the ERC-20 fee token (pre-approve and leave `msg.value = 0`) or in native ETH (`msg.value > 0`), in which case the gateway swaps it to the fee token via Uniswap V2 on your behalf. The fee is held against the commitment and released to the solver on a successful fill, or refunded to the user if the order is cancelled.

### Session key

The SDK generates a one-time session keypair per order and sets `order.session` to its public address. The session key authorizes exactly one solver to fill the order via an EIP-712 `SelectSolver` signature. This is handled automatically when you use `IntentGateway.execute()` or `IntentGateway.placeOrder()`.

### Commitment

After fee deduction and field stamping, the order commitment is:

```
commitment = keccak256(abi.encode(order))  // with reduced inputs, filled user/source/nonce
```

The SDK populates `order.id` with this commitment after the `OrderPlaced` event is parsed.

## Quoting Orders

Before constructing a USDC/cNGN or USDT/cNGN order, attach the Hyperbridge indexer and call `quoteIntent()`. Phantom snapshots are the default quote source, so no strategy option is needed. These pairs are priced from the latest directional `PhantomOrderPriceSnapshot`, which aggregates valid solver bids into a liquidity-weighted median.

The supported directions are:

- USDC â†’ cNGN
- cNGN â†’ USDC
- USDT â†’ cNGN
- cNGN â†’ USDT

```typescript lineNumbers title="quote-cngn.ts" icon="typescript"
import { createQueryClient, IntentGateway, type Order } from "@hyperbridge/sdk";
import { parseUnits, zeroAddress, zeroHash } from "viem";

const queryClient = createQueryClient({
  url: "https://nexus.indexer.polytope.technology",
});

const intentGateway = (
  await IntentGateway.create(sourceChain, destChain, coprocessor)
).withQueryClient(queryClient);

const quote = await intentGateway.quoteIntent({
  tokenIn: USDC_SOURCE,
  tokenOut: CNGN_DESTINATION,
  amountIn: parseUnits("100", 6),
});

// The gateway protocol fee is already reflected in quote.amountOut.
const minAmountOut = (quote.amountOut * (10_000n - SLIPPAGE_BPS)) / 10_000n;

const order: Order = {
  // user/source/nonce/session are populated by the SDK during placement
  user: zeroHash,
  source: sourceChain.config.stateMachineId,
  destination: destChain.config.stateMachineId,
  deadline,
  nonce: 0n,
  fees: 0n,
  session: zeroAddress,
  predispatch: { assets: [], call: "0x" },
  inputs: [{ token: USDC_SOURCE, amount: quote.amountIn }],
  output: {
    beneficiary,
    assets: [{ token: CNGN_DESTINATION, amount: minAmountOut }],
    call: "0x",
  },
};
```

### How the snapshot becomes a quote

Each snapshot records `standardAmount` of the input token and the liquidity-weighted median solver output as `medianPrice`. The SDK reads the newest snapshot for the exact direction on the canonical Base market and computes:

```
rate = medianPrice / standardAmount

exact input:
  netInput = amountIn âˆ’ floor(amountIn Ã— protocolFeeBps / 10_000)
  amountOut = floor(netInput Ã— medianPrice / standardAmount)

exact output:
  netInput = ceil(amountOut Ã— standardAmount / medianPrice)
  amountIn = netInput grossed up for the protocol fee
```

The reverse direction uses its own snapshot; the SDK never inverts the forward price because solver spreads and available liquidity can differ by direction. `quoteMetadata` includes the snapshot commitment, block number, timestamp, bid count, median/range, standard amount, and applied protocol fee for auditing or freshness checks.

If the indexer was not attached, or no valid directional snapshot exists, `quoteIntent()` throws. It never silently falls back to Uniswap; callers must explicitly request another quote strategy.

## Graffiti

Every `placeOrder` call accepts an optional `graffiti` field â€” a `bytes32` tag that identifies the frontend or integration that originated the order. It is passed as calldata and enables on-chain attribution of orderflow to specific integrators for revenue sharing.

Pass your graffiti as the second argument to `executeBest()` / `execute()`. The SDK exports `DEFAULT_GRAFFITI` â€” a zero `bytes32` â€” for unattributed orders.

```typescript lineNumbers title="graffiti.ts" icon="typescript"
import { DEFAULT_GRAFFITI } from "@hyperbridge/sdk"
import { stringToHex, padHex } from "viem"

// Encode your integrator identifier as a right-padded bytes32
const MY_GRAFFITI = padHex(stringToHex("my-app"), { size: 32 })

const gen = intentGateway.executeBest(order, MY_GRAFFITI, {
  auctionTimeMs: 15_000,
})

// Use DEFAULT_GRAFFITI to opt out of attribution
const gen = intentGateway.executeBest(order, DEFAULT_GRAFFITI, { auctionTimeMs: 15_000 })
```

## The `executeBest()` Generator

The SDK exposes two execution entry points:

- **`executeBest()`** â€” the batteries-included autopilot. It runs the full lifecycle and auto-selects the best bid each round (sort â†’ simulate â†’ sign â†’ submit). Use this when you want the SDK to pick the winning bid for you. This is the recommended starting point and the path documented below.
- **`execute()`** â€” the consumer-driven variant. Identical placement and auction flow, but instead of auto-selecting it hands you the bids and lets you rank and execute one yourself. See [Custom bid selection with `execute()`](#custom-bid-selection-with-execute) below.

Both share the same signature and the same `IntentOrderStatusUpdate` stream.

`executeBest()` auto-estimates fees, encodes and broadcasts the `placeOrder` transaction, runs a timed auction to collect solver bids, submits the winning `UserOperation`, and streams real-time status updates. During the auction period, the generator yields `NEW_BID` each time a previously unseen bid arrives, giving callers real-time visibility into the auction. After the auction ends, the best bid is selected and submitted. If a round's bids all fail, it keeps polling for fresh bids until the deadline. For same-chain orders it loops back after each partial fill, retrying until the order is fully filled or the deadline is reached.

### Generator protocol

The generator is bidirectional â€” it pauses at two points to hand control back to the caller:

**First yield** â€” `AWAITING_PLACE_ORDER`. The generator yields `{ status: "AWAITING_PLACE_ORDER", to, data, value, sessionPrivateKey }`. You sign and submit the transaction, then pass the result back via `gen.next(result)`. The result can be either:
- A **serialized signed transaction** (raw RLP-encoded bytes) â€” the SDK will broadcast it and wait for confirmation.
- A **transaction hash** (exactly 66 characters: `0x` + 64 hex chars) â€” the SDK will wait for the receipt of an already-submitted transaction.

**Second yield** â€” `ORDER_PLACED`. After the transaction is confirmed, the generator yields `{ status: "ORDER_PLACED", order, receipt }` where `order` is the finalized `Order` (with `nonce`, `id`, and `inputs` populated from the `OrderPlaced` event) and `receipt` is the full viem `TransactionReceipt` of the placement transaction.

**Subsequent yields** â€” status updates. The generator then yields `IntentOrderStatusUpdate` objects continuously until the order reaches a terminal state (`FILLED`, `EXPIRED`). During the auction period (`auctionTimeMs`), the generator yields `NEW_BID` each time a new solver bid is discovered â€” these are informational and do not affect bid selection. Once the auction ends, `BIDS_RECEIVED` is yielded with the full set of collected `Bid` objects, and `executeBest` selects and submits the best one. `FAILED` events are informational â€” the executor automatically retries by looping back to bid polling. Call `gen.next()` with no arguments to advance after each update.

```typescript lineNumbers title="executeBest.ts" icon="typescript"
import { IntentOrderStatus, DEFAULT_GRAFFITI } from "@hyperbridge/sdk"
import type { IntentOrderStatusUpdate, Order } from "@hyperbridge/sdk"


async function runOrder(order: Order) {
  const gen = intentGateway.executeBest(order, DEFAULT_GRAFFITI, {
    auctionTimeMs: 15_000, // duration in ms to collect bids before selecting the best one
    pollIntervalMs: 5_000, // how often to poll for new bids during the auction (default: SDK-defined)
  })

  // First yield: AWAITING_PLACE_ORDER â€” placeOrder calldata + estimated native fee
  const { value: { to, data, value } } = await gen.next()

  // Sign the placeOrder transaction. value covers the fee token swap if paying in native.
  const signedTx = await walletClient.signTransaction({
    to,
    data,
    value,
    type: "eip1559",
  })

  // Second yield: ORDER_PLACED â€” finalized order (nonce, id, inputs set from OrderPlaced event)
  const { value: { order, receipt } } = await gen.next(signedTx)
  console.log("placeOrder tx:", receipt.transactionHash)
  // store the full order incase you need to cancel it later
  console.log("Order commitment:", order.id)

  // then stream status updates
  for await (const update of gen) {
    switch (update.status) {
      case IntentOrderStatus.AWAITING_BIDS:
        // Auction started â€” SDK is polling the Hyperbridge coprocessor for solver bids
        console.log("Auction started, waiting for solver bidsâ€¦")
        break
      case IntentOrderStatus.NEW_BID:
        // A new bid arrived during the auction period (informational â€” bid selection happens after auction)
        console.log(`New bid during auction`)
        break
      case IntentOrderStatus.BIDS_RECEIVED:
        // Auction ended â€” all collected bids are available; executeBest picks the best one
        console.log(`Auction ended, ${update.bidCount} bids collected`)
        break
      case IntentOrderStatus.BID_SELECTED:
        // Best bid chosen and UserOp submitted to the ERC-4337 bundler
        // Cross-chain: the generator idles until the deadline â€” Hyperbridge finalisation is async
        // Same-chain: SDK waits for the fill tx, then moves to FILLED or PARTIAL_FILL
        console.log("Bid selected, solver:", update.selectedSolver, "tx:", update.transactionHash)
        break
      case IntentOrderStatus.PARTIAL_FILL: {
        // Same-chain only: a solver partially filled the output
        // SDK automatically loops back to AWAITING_BIDS for the remaining amount
        const { filledAssets, totalFilledAssets, remainingAssets } = update
        console.log("Partial fill:", filledAssets, "| total filled:", totalFilledAssets, "| remaining:", remainingAssets)
        break
      }
      case IntentOrderStatus.FILLED:
        // Order fully filled; escrow released to the solver
        console.log("Order fully filled! tx:", update.transactionHash)
        return
      case IntentOrderStatus.EXPIRED:
        // Order deadline reached, or no new bids arrived for the remaining amount
        // The order is still live on-chain â€” cancel it or wait for a solver to fill the rest manually
        console.warn("Order expired:", update.error, "Remaining:", update.remainingAssets)
        return
      case IntentOrderStatus.FAILED:
        // Retryable error (e.g. bundler rejection, simulation failure)
        // The executor automatically retries â€” this is informational only
        console.warn("Retryable error, retrying:", update.error)
        break
    }
  }
}
```


### Status reference

| Status | Trigger | Key fields |
|--------|---------|-----------|
| `AWAITING_PLACE_ORDER` | Before broadcasting â€” SDK yields calldata for signing | `to`, `data`, `value`, `sessionPrivateKey` |
| `ORDER_PLACED` | `placeOrder` tx confirmed on-chain | `order` (finalized `Order`), `receipt` |
| `AWAITING_BIDS` | Auction started, polling coprocessor for bids | `commitment`, `totalFilledAssets`, `remainingAssets` |
| `NEW_BID` | New bid discovered during auction (informational) | `commitment`, `bid` (one `Bid`) |
| `BIDS_RECEIVED` | Auction ended, all bids ready for selection | `commitment`, `bidCount`, `bids` (`Bid[]`) |
| `BID_SELECTED` | Best bid chosen and UserOp sent to bundler | `commitment`, `selectedSolver`, `userOpHash`, `userOp`, `transactionHash` |
| `PARTIAL_FILL` | Same-chain partial fill confirmed; loop restarts | `commitment`, `filledAssets`, `totalFilledAssets`, `remainingAssets` |
| `FILLED` | Order fully filled | `commitment`, `userOpHash`, `transactionHash` |
| `EXPIRED` | Deadline reached or no new bids available â€” terminal | `commitment`, `totalFilledAssets`, `remainingAssets`, `error` |
| `FAILED` | Retryable error (e.g. bundler rejection) â€” executor retries automatically | `commitment`, `totalFilledAssets`, `remainingAssets`, `error` |

### Options

Both `executeBest()` and `execute()` take the same options:

| Option | Type | Default | Description |
|--------|------|---------|-------------|
| `auctionTimeMs` | `number` | â€” | **Required.** Duration in ms to collect bids before selecting the best one. During this period, `NEW_BID` is yielded each time a new bid is discovered. After the auction ends, `BIDS_RECEIVED` is yielded. |
| `pollIntervalMs` | `number` | SDK default | How frequently (ms) to poll the coprocessor for new bids during and after the auction |
| `maxPriorityFeePerGasBumpPercent` | `number` | `8` | Bump percentage applied to the priority fee estimate during fee estimation |
| `maxFeePerGasBumpPercent` | `number` | `10` | Bump percentage applied to the max fee estimate during fee estimation |
| `solver` | `{ address, timeoutMs }` | â€” | Restrict bids to `address` until `timeoutMs` elapses, then accept any solver |

## Custom bid selection with `execute()`

`executeBest()` picks the winning bid for you. When you need your own ranking â€” preferred solvers, reputation, MEV protection, batching the fill with other calls â€” use `execute()` instead. It is identical to `executeBest()` except at the `BIDS_RECEIVED` stage: instead of auto-selecting, it hands you the collected `Bid` objects and waits for you to execute one and feed the result back.

`execute()` is **bidirectional in two places**: after `AWAITING_PLACE_ORDER` you feed back the signed placement tx (as with `executeBest()`), and after `BIDS_RECEIVED` you feed back the `SelectBidResult` returned by `bid.execute()` (or `undefined` to keep polling for fresh bids).

Each `Bid` exposes:

- `solverAddress`, `outputs`, `relayerFee`, `nativeDispatchFee`, `userOp` â€” metadata for ranking.
- `simulate()` â€” pre-check the bid off-chain via `eth_call`; throws if it would revert.
- `execute()` â€” sign the solver selection and submit the UserOp to the bundler; returns a `SelectBidResult`.
- `outputUsdValue()` â€” price the bid's outputs in USD (`null` if any token can't be priced).

```typescript lineNumbers title="custom-selection.ts" icon="typescript"
import { IntentOrderStatus, DEFAULT_GRAFFITI } from "@hyperbridge/sdk"
import type { Order, SelectBidResult, HexString } from "@hyperbridge/sdk"

async function runOrder(order: Order) {
  const gen = intentGateway.execute(order, DEFAULT_GRAFFITI, { auctionTimeMs: 15_000 })

  let input: HexString | SelectBidResult | undefined
  while (true) {
    const { value: update, done } = await gen.next(input)
    input = undefined
    if (done) break

    switch (update.status) {
      case IntentOrderStatus.AWAITING_PLACE_ORDER: {
        // Sign + broadcast the placeOrder tx, feed the hash back
        input = await walletClient.sendTransaction({ to: update.to, data: update.data, value: update.value })
        break
      }
      case IntentOrderStatus.BIDS_RECEIVED: {
        // Your selection logic. Here: highest USD output that passes simulation.
        const sorted = await intentGateway.sortBids(order, update.bids)
        for (const bid of sorted) {
          try {
            await bid.simulate()
            input = await bid.execute() // SelectBidResult fed back to the generator
            break
          } catch {
            // bid failed simulation/execution â€” try the next
          }
        }
        // leaving `input` undefined tells the executor to keep polling for fresh bids
        break
      }
      case IntentOrderStatus.FILLED:
        console.log("Order filled!")
        return
    }
  }
}
```

<Callout type="info">
After an app restart, resume a previously placed order with `resume()` (consumer-driven) or `resumeBest()` (autopilot). They take the same options as `execute()` minus the placement step, and require the order's `id` and `session`.
</Callout>

### `EXPIRED`

When `EXPIRED` fires, the order deadline has been reached or no new bids are available. The user can cancel and reclaim the remaining escrow. See [Cancelling Orders](./cancelling-orders) for cancellation options.

### `FAILED` (retryable)

`FAILED` events are informational â€” the executor automatically retries by looping back to bid polling. Common causes include bundler gas price rejections or bid simulation failures. The retry continues until the order reaches `FILLED` or `EXPIRED`.

## Same-Chain Orders

Set `destination` to the source chain's state machine ID. The contract verifies `keccak256(source) == keccak256(destination)` and settles atomically on-chain with no cross-chain messaging.

```typescript lineNumbers title="same-chain-order.ts" icon="typescript"
import { toHex, parseUnits } from "viem"

const USDC = "0xYourUsdcAddress" as `0x${string}`

const order: Order = {
  user: "0x",                                                   // set by contract (msg.sender)
  source: "0x",                                                 // set by contract (host().host())
  destination: toHex(sourceChain.config.stateMachineId),           // same as source â†’ same-chain
  deadline: BigInt(await sourceChain.client.getBlockNumber()) + 200n, // ~40 min on Ethereum
  nonce: 0n,                                                    // set by contract (_nonce++)
  fees: 0n,                                                     // auto-estimated by SDK
  session: "0x",                                                // set by SDK (generated keypair)
  predispatch: { assets: [], call: "0x" },
  inputs: [{ token: USDC, amount: parseUnits("1000", 6) }],
  output: {
    beneficiary: account.address,
    assets: [{ token: USDC, amount: parseUnits("990", 6) }],
    call: "0x",
  },
}

await runOrder(order)
```

<Callout type="info">
`TokenInfo.token` and `output.beneficiary` are standard 20-byte EVM addresses (`0xâ€¦`). The SDK left-pads them to `bytes32` when encoding the contract call â€” you do not need to pad them yourself.
</Callout>

### Partial Fills

Same-chain orders support partial fills. Multiple solvers can each fill a portion of the output independently; the contract releases escrowed input tokens proportionally per fill:

```
releasedEscrow[i] = inputs[i].amount Ã— fillAmount / totalRequired
```

The contract emits `PartialFill` for each partial fill and `OrderFilled` only when every output reaches its target. Postdispatch calldata (`output.call`) executes only on full fill, never on partials.

## Cross-Chain Orders

Set `destination` to the target chain's state machine ID. The contract verifies `keccak256(source) != keccak256(destination)` and routes to cross-chain logic.

Cross-chain fills are **all-or-nothing** â€” the solver must provide the full required amount for every output asset in a single transaction. Partial fills are not supported for cross-chain orders. On fill, the contract dispatches a `RedeemEscrow` POST request via Hyperbridge back to the source chain, which releases escrowed tokens to the solver on receipt.

`execute()` exits after `BID_SELECTED` for cross-chain orders â€” it does not wait for Hyperbridge finalisation. To track cross-chain settlement, monitor the `EscrowReleased` event on the source chain or use the indexer.

```typescript lineNumbers title="cross-chain-order.ts" icon="typescript"
import { toHex, parseUnits } from "viem"

const USDC_SOURCE = "0xUsdcOnEthereum" as `0x${string}`
const USDC_DEST   = "0xUsdcOnArbitrum" as `0x${string}`

const order: Order = {
  user: "0x",
  source: "0x",
  destination: toHex("EVM-42161"),                     // Arbitrum
  deadline: BigInt(await sourceChain.client.getBlockNumber()) + 200n,
  nonce: 0n,
  fees: 0n,
  session: "0x",
  predispatch: { assets: [], call: "0x" },
  inputs:  [{ token: USDC_SOURCE, amount: parseUnits("1000", 6) }],
  output: {
    beneficiary: account.address,                               // recipient on Arbitrum
    assets: [{ token: USDC_DEST, amount: parseUnits("990", 6) }],
    call: "0x",
  },
}

await runOrder(order)
```

### Surplus sharing

When a solver overpays on a cross-chain fill, the excess is split:

```
protocolShare    = surplus Ã— surplusShareBps / 10_000
beneficiaryShare = surplus âˆ’ protocolShare
```

If the order has postdispatch calldata, **100% of surplus goes to the protocol** to prevent manipulation of calldata side effects.

## Predispatch Calldata

Predispatch runs arbitrary calls *before* tokens are escrowed. Assets listed in `predispatch.assets` are transferred from the caller to the `CallDispatcher`, the encoded `Call[]` in `predispatch.call` execute against it, and then the contract measures the `CallDispatcher`'s resulting balance for each input token â€” escrowing exactly `inputs[i].amount` and sweeping any excess as dust.

The canonical use case is **swap-then-escrow**: the user sends WETH, the predispatch call swaps it to USDC via a DEX, and the USDC balance that lands in the `CallDispatcher` becomes the escrowed order input. The user never needs to hold the input token before placing the order.

```solidity
struct Call {
    address to;
    uint256 value;
    bytes data;
}
```

```typescript lineNumbers title="predispatch-order.ts" icon="typescript"
import { toHex, parseEther, parseUnits, parseAbi, encodeAbiParameters, encodeFunctionData } from "viem"
import { IntentGatewayV2ABI } from "@hyperbridge/sdk"

const WETH        = "0xWethAddress" as `0x${string}`
const USDC        = "0xUsdcAddress" as `0x${string}`
const ROUTER      = "0xUniswapV2Router" as `0x${string}`

// Fetch the CallDispatcher address from the gateway params
const { dispatcher } = await sourceChain.client.readContract({ address: "0xGatewayAddress", abi: IntentGatewayV2ABI, functionName: "params" })

const wethIn      = parseEther("1")
const usdcMin     = parseUnits("2000", 6)
const swapDeadline = BigInt(Math.floor(Date.now() / 1000) + 600)

function encodeCalls(calls: { to: `0x${string}`; value: bigint; data: `0x${string}` }[]) {
  return encodeAbiParameters(
    [{ type: "tuple[]", components: [{ name: "to", type: "address" }, { name: "value", type: "uint256" }, { name: "data", type: "bytes" }] }],
    [calls],
  ) as `0x${string}`
}

const erc20Abi  = parseAbi(["function approve(address,uint256) returns (bool)"])
const routerAbi = parseAbi(["function swapExactTokensForTokens(uint256,uint256,address[],address,uint256) returns (uint256[])"])

// 1. Approve the router to spend WETH from inside the CallDispatcher
const approveData = encodeFunctionData({ abi: erc20Abi, functionName: "approve", args: [ROUTER, wethIn] })
// 2. Swap WETH â†’ USDC; output lands in the CallDispatcher
const swapData    = encodeFunctionData({ abi: routerAbi, functionName: "swapExactTokensForTokens", args: [wethIn, usdcMin, [WETH, USDC], dispatcher, swapDeadline] })

const order: Order = {
  user: "0x",
  source: "0x",
  destination: toHex("EVM-42161"),
  deadline: BigInt(await sourceChain.client.getBlockNumber()) + 200n,
  nonce: 0n,
  fees: 0n,
  session: "0x",
  predispatch: {
    assets: [{ token: WETH, amount: wethIn }],               // transferred to CallDispatcher first
    call: encodeCalls([
      { to: WETH, value: 0n, data: approveData },
      { to: ROUTER, value: 0n, data: swapData },
    ]),
  },
  inputs: [{ token: USDC, amount: usdcMin }],                 // escrowed after the swap
  output: {
    beneficiary: account.address,
    assets: [{ token: USDC, amount: usdcMin }],
    call: "0x",
  },
}

await runOrder(order)
```

<Callout type="warn">
The gateway escrows exactly `inputs[i].amount`. Any USDC left in the `CallDispatcher` above that amount is swept into the gateway as dust and emits `DustCollected`. Size `usdcMin` conservatively to avoid over-escrowing.
</Callout>

## Postdispatch Calldata

Postdispatch runs calls *after* the solver delivers output tokens. Set `output.beneficiary` to the `CallDispatcher` so it receives the tokens, then encode `output.call` with what should happen next.

**Timing differences by order mode:**

| Mode | When postdispatch executes |
|------|---------------------------|
| Same-chain | Only after the order is **fully** filled â€” partial fills do not trigger it |
| Cross-chain | Immediately after the solver transfers tokens, **before** the `RedeemEscrow` message is dispatched |

After execution, any tokens remaining on the `CallDispatcher` are swept back to the gateway as dust. When `output.call` is non-empty, **100% of any surplus goes to the protocol** â€” `surplusShareBps` is not applied.

```typescript lineNumbers title="postdispatch-order.ts" icon="typescript"
import { toHex, parseUnits, parseAbi, encodeAbiParameters, encodeFunctionData } from "viem"
import { IntentGatewayV2ABI } from "@hyperbridge/sdk"

const USDC      = "0xUsdcAddress" as `0x${string}`
const AAVE_POOL = "0xAavePoolAddress" as `0x${string}`
const { dispatcher } = await sourceChain.client.readContract({ address: "0xGatewayAddress", abi: IntentGatewayV2ABI, functionName: "params" })

const outputAmount = parseUnits("990", 6)

function encodeCalls(calls: { to: `0x${string}`; value: bigint; data: `0x${string}` }[]) {
  return encodeAbiParameters(
    [{ type: "tuple[]", components: [{ name: "to", type: "address" }, { name: "value", type: "uint256" }, { name: "data", type: "bytes" }] }],
    [calls],
  ) as `0x${string}`
}

const erc20Abi = parseAbi(["function approve(address,uint256) returns (bool)"])
const aaveAbi  = parseAbi(["function supply(address,uint256,address,uint16)"])

// Approve Aave to pull USDC from the CallDispatcher, then deposit on behalf of the user
const approveData = encodeFunctionData({ abi: erc20Abi, functionName: "approve", args: [AAVE_POOL, outputAmount] })
const supplyData  = encodeFunctionData({ abi: aaveAbi,  functionName: "supply",  args: [USDC, outputAmount, account.address, 0] })

const order: Order = {
  user: "0x",
  source: "0x",
  destination: toHex("EVM-42161"),
  deadline: BigInt(await sourceChain.client.getBlockNumber()) + 200n,
  nonce: 0n,
  fees: 0n,
  session: "0x",
  predispatch: { assets: [], call: "0x" },
  inputs: [{ token: USDC, amount: parseUnits("1000", 6) }],
  output: {
    beneficiary: dispatcher,                                   // CallDispatcher receives output tokens
    assets: [{ token: USDC, amount: outputAmount }],
    call: encodeCalls([
      { to: USDC, value: 0n, data: approveData },
      { to: AAVE_POOL, value: 0n, data: supplyData },
    ]),
  },
}

await runOrder(order)
```
