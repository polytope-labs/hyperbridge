# Hyper Fungible Token

HyperFungibleToken is a cross-chain token standard where each token deployment is its own bridge application. There is no shared custody pool, no governance overhead, and no token-governor — the token contract itself handles cross-chain messaging via [ISMP](/protocol/ismp).

## How It Works

There are two contract types, each suited to different use cases:

### WrappedHyperFungibleToken (Lock/Unlock)

Deployed on the token's **home chain** — the chain where the canonical ERC20 supply lives. The wrapper custodies the underlying token without minting any new supply.

When a user bridges out, the wrapper locks the underlying tokens and dispatches an ISMP POST request. When a user bridges back, the wrapper unlocks and returns the underlying tokens to the recipient. On timeout, the locked tokens are returned to the sender.

For WETH wrappers, users can send native ETH directly — the contract wraps it via `WETH.deposit()`. On timeout, the contract unwraps and refunds native ETH.

### HyperFungibleToken (Burn/Mint)

Deployed on every **remote chain**. The token inherits from ERC20 and [HyperApp](/developers/evm/api/hyperapp), so it is both the token and the bridge in one contract.

When tokens arrive from the home chain, the contract mints new supply to the recipient. When a user bridges back, it burns their tokens and dispatches an ISMP POST request to the home chain, where the WrappedHFT unlocks the underlying. On timeout, tokens are re-minted to the original sender.

This architecture keeps canonical supply on the home chain. The WrappedHFT accumulates locked tokens as users bridge out, and releases them when users bridge back. No pre-funding is required.

### Cross-Chain Message Format

Both contracts encode the same ABI message body, ensuring full interoperability between `HyperFungibleToken`, `WrappedHyperFungibleToken`, and [`pallet-hyper-fungible-token`](/developers/polkadot/hyper-fungible-token):

```solidity lineNumbers
struct Message {
    bytes from;     // original sender (for timeout refunds)
    bytes to;       // recipient on destination chain
    uint256 amount; // token amount
    bytes data;     // optional calldata for CallDispatcher
}
```

### CallDispatcher

Both contracts support optional calldata execution on the destination chain. If the `data` field in `SendParams` is non-empty, the contract calls `ICallDispatcher.dispatch(data)` after minting/unlocking tokens. This enables composable cross-chain interactions like transfer-and-swap or transfer-and-stake.

### ERC165 Interface Detection

Both contracts implement ERC165 for on-chain type detection:

| Contract | Interface ID |
|----------|-------------|
| `IHyperFungibleToken` | `0x7200c457` |
| `IWrappedHyperFungibleToken` | `0x903d5332` |

The [TypeScript SDK](/developers/sdk/hyper-fungible-token) uses this to automatically determine which approval flow is needed.

## Deploying a WrappedHyperFungibleToken

Deploy this on the token's home chain where the canonical ERC20 supply exists.

<Steps>
<Step>
### Deploy the wrapper with CREATE2

Using CREATE2 ensures the same contract address across all chains, simplifying peer configuration and frontend integration. Use a deterministic deployer like [CreateX](https://github.com/pcaversaccio/createx) or OpenZeppelin's `Create2` library.

```solidity lineNumbers
import {WrappedHyperFungibleToken} from "@hyperbridge/core/apps/WrappedHyperFungibleToken.sol";
import {IHyperFungibleToken} from "@hyperbridge/core/interfaces/IHyperFungibleToken.sol";

bytes32 salt = keccak256("my-token-v1");
WrappedHyperFungibleToken wrapper = new WrappedHyperFungibleToken{salt: salt}();
```
</Step>
<Step>
### Set up a CallDispatcher (optional)

If you want to support calldata execution on the destination chain (e.g. transfer-and-swap), use an existing `CallDispatcher` deployment — addresses are listed on the [contract addresses](/developers/evm/contract-addresses/mainnet) page.
</Step>
<Step>
### Configure the wrapper

Set the ISMP host, CallDispatcher, the underlying ERC20 token, and whether it's a WETH contract. The host address is immutable after the first `configure` call.

```solidity lineNumbers
import {IWrappedHyperFungibleToken} from "@hyperbridge/core/interfaces/IHyperFungibleToken.sol";

IWrappedHyperFungibleToken(address(wrapper)).configure(
    IWrappedHyperFungibleToken.WrappedConfigOptions({
        host: ISMP_HOST_ADDRESS,
        dispatcher: CALL_DISPATCHER_ADDRESS,
        underlying: USDC_ADDRESS,
        isWeth: false
    })
);
```

Find the `IsmpHost` address for your chain on the [contract addresses](/developers/evm/contract-addresses/mainnet) page.

**WETH example:**

```solidity lineNumbers
IWrappedHyperFungibleToken(address(wrapper)).configure(
    IWrappedHyperFungibleToken.WrappedConfigOptions({
        host: ISMP_HOST_ADDRESS,
        dispatcher: CALL_DISPATCHER_ADDRESS,
        underlying: WETH_ADDRESS,
        isWeth: true
    })
);
```

When `isWeth` is `true`, users can send native ETH via `msg.value` and the contract wraps it automatically. On timeout, the contract unwraps WETH and refunds native ETH.
</Step>
<Step>
### Register peer chains

Each contract must know the address of its peer on every destination chain. **Both sides must register each other** — this is how the contracts verify the authenticity of incoming cross-chain messages.

```solidity lineNumbers
import {StateMachine} from "@hyperbridge/core/libraries/StateMachine.sol";

// Register the HFT peer on Arbitrum
IHyperFungibleToken(address(wrapper)).addChain(
    StateMachine.evm(42161),
    abi.encodePacked(arbitrumHftAddress)
);
```

For substrate chain peers, the module ID is the pallet's 8-byte ID:

```solidity lineNumbers
IHyperFungibleToken(address(wrapper)).addChain(
    StateMachine.polkadot(paraId),
    abi.encodePacked(bytes8("pall_hft"))
);
```
</Step>
</Steps>

## Deploying a HyperFungibleToken

Deploy this on every remote chain where the token doesn't have native supply.

<Steps>
<Step>
### Deploy the token with CREATE2

Use the same salt as the home chain deployment to get a predictable address.

```solidity lineNumbers
import {HyperFungibleToken} from "@hyperbridge/core/apps/HyperFungibleToken.sol";
import {IHyperFungibleToken} from "@hyperbridge/core/interfaces/IHyperFungibleToken.sol";

bytes32 salt = keccak256("my-token-v1");
HyperFungibleToken token = new HyperFungibleToken{salt: salt}("Wrapped USDC", "wUSDC");
```

The deployer becomes the contract owner. The HFT has no initial supply — tokens are only minted when cross-chain transfers arrive.
</Step>
<Step>
### Configure the contract

```solidity lineNumbers
IHyperFungibleToken(address(token)).configure(
    IHyperFungibleToken.ConfigOptions({
        host: ISMP_HOST_ADDRESS,
        dispatcher: CALL_DISPATCHER_ADDRESS
    })
);
```
</Step>
<Step>
### Register peer chains

Register the WrappedHFT on the home chain and any other HFT peers:

```solidity lineNumbers
import {StateMachine} from "@hyperbridge/core/libraries/StateMachine.sol";

// Register the WrappedHFT on Ethereum (home chain)
IHyperFungibleToken(address(token)).addChain(
    StateMachine.evm(1),
    abi.encodePacked(ethereumWrapperAddress)
);

// Register the HFT peer on Base
IHyperFungibleToken(address(token)).addChain(
    StateMachine.evm(8453),
    abi.encodePacked(baseHftAddress)
);
```

On each peer chain, do the reverse — register this chain's HFT address.
</Step>
</Steps>

## Sending Tokens

Once deployed and configured, users call `send()` to initiate a cross-chain transfer:

```solidity lineNumbers
import {StateMachine} from "@hyperbridge/core/libraries/StateMachine.sol";
import {IHyperFungibleToken} from "@hyperbridge/core/interfaces/IHyperFungibleToken.sol";

// Pay fees in native token
IHyperFungibleToken(tokenAddress).send{value: nativeFee}(
    IHyperFungibleToken.SendParams({
        dest: StateMachine.evm(42161),
        to: abi.encodePacked(recipientAddress),
        amount: 1 ether,
        timeout: 3600,
        relayerFee: relayerFee,
        data: ""
    })
);
```

To pay fees in the host's fee token instead of native, send with `msg.value = 0`. The contract will pull fee tokens from the caller via `transferFrom`:

```solidity lineNumbers
// Approve fee token first
IERC20(feeToken).approve(tokenAddress, feeAmount);

// Pay fees in fee token (no msg.value)
IHyperFungibleToken(tokenAddress).send(
    IHyperFungibleToken.SendParams({
        dest: StateMachine.evm(42161),
        to: abi.encodePacked(recipientAddress),
        amount: 1 ether,
        timeout: 3600,
        relayerFee: relayerFee,
        data: ""
    })
);
```

### Fee Estimation

Use the inherited [HyperApp](/developers/evm/api/hyperapp) `quote()` and `quoteNative()` methods to estimate fees on-chain. For a higher-level experience, use the [TypeScript SDK](/developers/sdk/hyper-fungible-token) which handles gas estimation, cross-chain fee token conversion, and decimal scaling automatically.

## Pausability

Both contracts inherit from OpenZeppelin's `Pausable`. The owner can pause and unpause all operations:

```solidity lineNumbers
IHyperFungibleToken(tokenAddress).pause();
IHyperFungibleToken(tokenAddress).unpause();
```

When paused, `HyperFungibleToken` blocks all ERC20 transfers (`transfer` and `transferFrom`) in addition to cross-chain `send()` and `onAccept()`. This is a full stop on all token movement — use it for emergency situations or during migrations.

## Security Model

### Source Verification

Every incoming message is verified against the registered peer:

```solidity lineNumbers
bytes memory expectedSource = _supportedChains[request.source];
if (keccak256(request.from) != keccak256(expectedSource)) revert UnauthorizedSource();
```

Only messages from registered peer contracts are accepted. An attacker cannot spoof messages from unregistered chains or impersonate the peer contract address on a registered chain — ISMP provides cryptographic proof that the message was actually dispatched on the source chain.

### Host Authorization

All ISMP callbacks (`onAccept`, `onPostRequestTimeout`) are protected by the `onlyHost` modifier inherited from [HyperApp](/developers/evm/api/hyperapp). Only the ISMP host contract can call these functions.

### Set-Once Host

The host address can only be set once via `configure()`. After the first call, subsequent calls cannot change the host — only the dispatcher can be updated.
