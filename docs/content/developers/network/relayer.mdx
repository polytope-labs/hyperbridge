# Running a Hyperbridge Relayer

The Hyperbridge relayer (`tesseract`) is a single binary that
drives every relaying responsibility a counterparty-chain operator has.
It replaces the older two-binary split between `tesseract` (messaging) and
`tesseract-consensus` (consensus), with the same responsibilities plus a
new **outbound** task that delivers Hyperbridge → chain proofs.

Everything a relayer operator used to do with the two binaries still
happens here — inbound messaging, inbound consensus (now opt-in per
chain via a `[<chain>.consensus]` sub-table), fee accumulation, and
automatic fee withdrawal. The difference is that it now runs in one
process and also fans out outbound `(consensus + messages)` submissions
from Hyperbridge to every configured destination.

## Prebuilt binaries

You can install a prebuilt binary for the Tesseract relayer with the following bash script

```bash lineNumbers
LATEST_TAG=$(curl -s https://hub.docker.com/v2/repositories/polytopelabs/tesseract/tags\?page_size\=1\&page\=2 | jq -r '.results[0].name')
wget -q --show-progress https://github.com/polytope-labs/hyperbridge/releases/download/tesseract-$LATEST_TAG/tesseract-x86_64-unknown-linux-gnu.tar.gz
tar -xvzf tesseract-x86_64-unknown-linux-gnu.tar.gz
# copy to $PATH
cp tesseract-x86_64-unknown-linux-gnu/tesseract $HOME/.local/bin/
```

or a 1-liner

```bash lineNumbers
LATEST_TAG=$(curl -s https://hub.docker.com/v2/repositories/polytopelabs/tesseract/tags\?page_size\=1\&page\=2 | jq -r '.results[0].name')
curl --proto '=https' --tlsv1.2 -LsSf https://github.com/polytope-labs/hyperbridge/releases/download/tesseract-$LATEST_TAG/tesseract-installer.sh | sh
```

## Running the relayer

The tesseract relayer command line interface expects two arguments, which are the paths to its configuration file and database file. If you're running it in a docker container, you will need to [map the directory](https://docs.docker.com/storage/bind-mounts/#start-a-container-with-a-bind-mount) where you've stored the configuration files on your host into the docker container. Tesseract will write its database to that directory once it's initialized.

```bash lineNumbers
docker run -d \
  --name=tesseract \
  --network=host \
  --restart=always \
  --volume=/dir/on/host:/home/root \
  polytopelabs/tesseract:latest \
  --config=/home/root/config.toml \
  --db=/home/root/tesseract.db
```

Access its logs using

```bash lineNumbers
docker logs -f tesseract
```

The relayer also exposes two one-shot subcommands:

- `log-consensus-state <STATE_MACHINE>` — fetch and print the initial
  ConsensusState for a chain (hex-encoded). Useful for seeding a fresh
  deployment.
- `withdraw` — run a single fee-withdrawal pass over every configured
  destination and exit. See [Withdrawing Fees](#withdrawing-fees).

### System Requirements

At the minimum, the hyperbridge relayer should be run on a machine with at least 4GB of RAM and a quad-core cpu. This relayer should also have at least a 100Mb/s connection if it is to query nodes over the internet.

## Configuration

The configuration file is a toml file that at the moment, that expects the following configuration options:

```toml lineNumbers
# Hyperbridge config, required
# Flat SubstrateConfig — the BEEFY prover runs as a separate operator-side
# binary; this relayer only consumes accepted proofs from the Hyperbridge
# node's offchain storage.
[hyperbridge]
# For mainnet "POLKADOT-3367"
state_machine = "KUSAMA-4009"

hashing = "Keccak"

# Hyperbridge node ws rpc endpoint.
# The node MUST expose offchain_localStorageGet (typically via
# --rpc-methods=Unsafe) so the outbound relayer can read accepted
# BEEFY consensus proofs.
rpc_ws = "ws://127.0.0.1:9944" # example endpoint

# hex-encoded private key for the relayer account on Hyperbridge.
# This is the private key to the Hyperbridge account where you earn BRIDGE token rewards
signer = ""

# Sets the maximum size of an rpc request or response
#  in bytes defaults to 150mb
# max_rpc_payload_size = 150000000

# (Optional) initial height
# This sets the height at which to start querying messages,
# This exists for testing and development
# misuse can cause the rpc to be overloaded with queries
# initial_height = 1000

# (Optional) Frequency in seconds to poll the chain
# for new state machine update events
# poll_interval = 10

# (Optional) Hyperbridge's consensus state id
# Only add this if relaying to a solochain.
# Relaying between solochains and parachains with the
# same relayer instance is not supported.
# For Paseo Testnet: PAS0
# For Polkadot Mainnet: DOT0
consensus_state_id = "PAS0"

# Relayer config, required
[relayer]
# Define your profitability configuration in basis points.
# 0 -> 0% i.e relay all requests, even unprofitable ones.
# 100 -> 1%. ie fees provided for requests must be
# profitable by at least 1%. etc.
minimum_profit_percentage = 100

# How frequently to initiate withdrawals in seconds
# defaults to 24 hours
withdrawal_frequency = 86400

# Minimum amount to withdraw when auto-withdrawing
# defaults to $100
minimum_withdrawal_amount = 100

# (Optional)
# If not empty, will filter requests to be delivered
# by originating module identifier (eg contract address/pallet ids)
# module_filter = []

# (Optional)
# How frequently to retry unprofitable or
# failed messages in seconds.
# If this is value not supplied retries will not be enabled.
# unprofitable_retry_frequency = 600

# (Optional) Opt this relayer into fisherman duty. When true, a task is
# spawned per chain that watches HB ↔ destination for byzantine
# state-machine updates and dispatches veto extrinsics.
# fisherman = false

# (Optional) Disable fee accumulation task
# If this value is true, fee accumulation will be disabled.
# This should only be enabled when running a private relayer
# disable_fee_accumulation = false

# (Optional) Deliver messages even when their estimate-gas simulation
# reports failure.
# deliver_failed = false

# (Optional) Per-(state_machine, max_interval_secs) entries enabling the
# passive liveness monitor. When set, a background task periodically
# checks how stale Hyperbridge's view of each listed chain is (inbound
# consensus side), and for substrate destinations also checks how stale
# the chain's view of Hyperbridge is (outbound HB -> substrate consensus
# side). If any update lags by more than its max_interval, the relayer
# process exits so an external supervisor (docker, systemd, k8s) can
# restart it. EVM destinations skip the outbound check, since their HB
# consensus path runs through the BEEFY proofs pipeline.
# maximum_update_intervals = [
#     [{ state_id = "EVM-1",       consensus_state_id = "ETH0" }, 900],
#     [{ state_id = "EVM-100",     consensus_state_id = "GNO0" }, 420],
#     [{ state_id = "EVM-56",      consensus_state_id = "BSC0" }, 300],
#     [{ state_id = "POLKADOT-2100", consensus_state_id = "DOT0" }, 600],
# ]

# Here you'll declare a new chain entry
# for every chain you want to support.

# Configuration for EVM chains
[ethereum]
type = "evm"

# State machine identifier for this evm chain.
# Must be specified as:
# - "EVM-{chainId}"
# The possible values:
# state_machine = "EVM-1" # Ethereum Mainnet
# state_machine = "EVM-11155111" # Ethereum Sepolia
# state_machine = "EVM-42161" # Arbitrum Mainnet
# state_machine = "EVM-421614" # Arbitrum Sepolia
# state_machine = "EVM-10" # Optimism Mainnet
# state_machine = "EVM-11155420" # Optimism Sepolia
# state_machine = "EVM-8453" # Base Mainnet
# state_machine = "EVM-84532" # Base Sepolia
# state_machine = "EVM-56" # Binance smart chain Mainnet
# state_machine = "EVM-97" # Binance smart chain Testnet
#
# (Optional) Both state_machine and ismp_host auto-derive at startup
# from eth_chainId + the relayer's built-in registry. Set either
# explicitly to override.
state_machine = "EVM-11155111"

# http(s) rpc urls for evm based chains
# Multiple rpc endpoints supported for increased reliability
rpc_urls = ["http://127.0.0.1:8545", "http://localhost:8545"]

# The consensus state identifier for this chain on hyperbridge.
# For Ethereum and its L2s this will be "ETH0"
# for BSC this will be "BSC0".
consensus_state_id = "ETH0"

# The IsmpHost contract address on this chain
ismp_host = ""

# (Optional)
# Maximum block range that should be used to query eth_getLogs
# in a single rpc call. if this parameter is not supplied,
# the range will not be split into smaller chunks
# and will be queried at once. This might be needed
# if you encounter rate limits from your rpc provider.
# query_batch_size = 1000

# (Optional) Hex-encoded private key for the relayer account on this chain.
# Setting `signer` opts the chain into the HB->chain outbound fan-out (and
# the related fee-withdrawal and fisherman roles), since those are the
# tasks that need to submit transactions on this chain. Omit `signer`
# to run this chain in inbound-only mode: events are read and pushed to
# Hyperbridge using the [hyperbridge] signer, but no transactions are
# submitted on this chain.
signer = ""

# (Optional)
# Maximum number of concurrent requests to be used for tx tracing.
# If not supplied will trace entire tx batches at once.
# This increases tracing speeds by tracing txs concurrently
#
# Note: Your rpc node must support `debug_traceCall`.
# If using an rpc provider, ensure that they support a
# large number of `debug_traceCall` requests/sec.
#
# Note: if you use the same provider for all your endpoints
# then a very high request/sec threshold is required (> 1000 / sec)
#
# eg 5 transactions would be traced concurrently on this client
# tracing_batch_size = 5

# (Optional)
# Buffer to add to gas prices in basis points
# to increase likelihood of the transactions
# going through e.g 100 (1%), 200 (2%), 500 (5%)
# gas_price_buffer = 100

# (Optional)
# The execution client implementation,
# only Geth or Erigon are fully supported
# The possible values: "Geth" | "Erigon"
# If this field is not set, the default is "Geth"
# client_type = "Geth"

# (Optional) initial height
# This sets the height at which to start querying messages,
# This exists for testing and development
# misuse can cause the rpc to be overloaded with queries
# initial_height = 1000

# (Optional) Frequency in seconds to poll the chain
# for new state machine update events
# poll_interval = 10

# (Optional) Opt this chain into inbound consensus relaying.
# Presence of [<chain>.consensus] is the sole signal to spawn an
# inbound-consensus task for the chain. Host fields (rpc_urls,
# signer, state_machine, ismp_host, consensus_state_id, ...) inherit
# from the parent block — don't re-declare them.
# [ethereum.consensus]
# type = "ethereum"
#
# [ethereum.consensus.host]
# beacon_http_urls = ["http://localhost:3000"]
# consensus_update_frequency = 60

# Substrate chain configurations
[substrate]
type = "substrate"

# The state machine identifier for this substrate based chain.
# must be one of:
#  "POLKADOT-{paraId}"
#  "KUSAMA-{paraId}"
#  "SUBSTRATE-{customId}"
# (Optional) state_machine auto-derives at startup via the
# IsmpRuntimeApi_host_state_machine runtime call; set explicitly to
# override.
state_machine = "KUSAMA-4009"

# Hashing can be "Keccak" or "Blake2"
hashing = "Blake2"

# substrate node ws(s) rpc endpoint.
rpc_ws = "ws://127.0.0.1:9944" # example endpoint

# The consensus state identifier for this chain on hyperbridge.
# "DOT0"  for parachains on mainnet and "PAS0" on Paseo testnet
consensus_state_id = "PAS0"

# (Optional)
# Configures the maximum size of an rpc request/response in bytes
# max_rpc_payload_size = 150000000

# (Optional) Hex-encoded private key for the relayer account on this chain.
# Same toggle semantics as the EVM example above: presence opts the chain
# into outbound, fee withdrawal, and fisherman; omitting it keeps the
# chain inbound-only.
signer = ""

# (Optional) Frequency in seconds to poll the chain
# for new state machine update events
# poll_interval = 10

# (Optional) initial height
# This sets the height at which to start querying messages,
# This exists for testing and development,
# misuse can cause the rpc to be overloaded with queries
# initial_height = 1000
# (Optional)
# This provides the relayer with the precision for the fee token on this substrate chain
# Needed for ensuring correct delivery fee estimation
# It defaults to 6 which is the decimal for USDC and USDT on Polkadot parachains
# fee_token_decimals = 6
```

It is optional to provide the configuration option for any of the connected chains, The only consequence is your relayer will not deliver requests from the ommitted chain as it has no way of querying the associated fees for requests originating from this chain.

### Inbound consensus (opt-in per chain)

Before consolidation, inbound-consensus relaying lived in a separate
`tesseract-consensus` binary. In this relayer it's opt-in **per chain**:
add a `[<chain>.consensus]` sub-table to that chain's block. Host
essentials (`rpc_urls`, `signer`, `state_machine`, `ismp_host`, parent
`consensus_state_id`) are inherited from the parent and don't need to be
re-declared. The sub-table only carries the consensus client's own
settings.

Chains **without** a `[<chain>.consensus]` sub-table still receive inbound
messaging and Hyperbridge → chain outbound delivery.

Concrete, paste-ready `[<chain>]` blocks for each supported chain (both
the messaging side and the consensus sub-table where applicable) live in
[Per-chain configurations](#per-chain-configurations) below.

### Outbound (Hyperbridge to chain)

Unlike the two-binary split, this relayer also handles outbound
delivery from Hyperbridge back to every configured destination.
One task subscribes to `pallet-beefy-consensus-proofs::ProofAccepted`
events on Hyperbridge. On each event it fetches the accepted proof from
Hyperbridge's offchain storage and fans out a batched
`(consensus + messages)` submission to every chain that has a `signer`
configured.

Authority-set rotations always propagate. Messaging-only proofs skip
chains with no pending messages. Lagging destinations are caught up
first by comparing BEEFY `current_authorities.id` against Hyperbridge's
and submitting any missing rotation proofs in chunks of **3 consensus
messages per batch** to stay inside EVM calldata and gas limits.

To opt a chain out of the outbound fan-out, omit `signer` from that
chain's block. Without a signer the relayer cannot submit transactions
on that chain, so it is automatically excluded from outbound delivery,
fee-withdrawal POSTs, and fisherman vetoes. Inbound messaging and
inbound consensus (if `[<chain>.consensus]` is set) continue to work,
since both submit on Hyperbridge using the `[hyperbridge]` signer.

## Per-chain configurations

The generic `[ethereum]` and `[substrate]` blocks in the reference
configuration above show every field with its comments. The sections
below give paste-ready blocks for each officially supported chain,
exactly as they appear in the consolidated `config.toml`. Each block
contains the messaging `[<chain>]` table and the `[<chain>.consensus]`
sub-table for inbound consensus relaying joined together. Drop the
blocks you need into your config after the `[hyperbridge]` and
`[relayer]` headers.

Replace `SIGNER-KEY` with the hex-encoded private key for the relayer
account on that chain, and replace RPC URLs with endpoints you trust.
`state_machine` is required per chain since the relayer keys its
routing table by it. `ismp_host` and the messaging-side
`consensus_state_id` auto-derive at startup from `eth_chainId` plus
the relayer's built-in registry (and from the
`IsmpRuntimeApi_host_state_machine` runtime call for substrate
chains), so they do not appear in the blocks. If you need to pin a
specific `ismp_host` by hand, the current values are in the
[mainnet](/developers/evm/contract-addresses/mainnet) and
[testnet](/developers/evm/contract-addresses/testnet) address tables.
Setting `signer` opts the chain into the Hyperbridge → chain outbound
fan-out (and the fee-withdrawal / fisherman roles); omit it to keep the
chain inbound only. Host fields (`rpc_urls`, `signer`, `state_machine`,
`ismp_host`, `consensus_state_id`) in the `[<chain>.consensus]`
sub-table are inherited from the parent messaging block and aren't
re-declared.

### Ethereum

Ethereum mainnet uses the sync-committee consensus client. For Sepolia,
swap `type` in the consensus sub-table to `"sepolia"`.

```toml lineNumbers
[ethereum]
type = "evm"
rpc_urls = ["http://localhost:3000"]
consensus_state_id = "ETH0"
signer = "SIGNER-KEY"
[ethereum.consensus]
type = "ethereum"
beacon_http_urls = ["http://localhost:3000"]
consensus_update_frequency = 60
```

### Arbitrum

Arbitrum and other Arbitrum Orbit L2s use the `arbitrum_orbit` consensus
client and read L2 state through an Ethereum L1 RPC. The `rollup_core`
address is chain-specific.

```toml lineNumbers
[arbitrum]
type = "evm"
rpc_urls = ["http://localhost:3100"]
consensus_state_id = "ARB0"
signer = "SIGNER-KEY"
[arbitrum.consensus]
type = "arbitrum_orbit"
ethereum_rpc_url = ["http://localhost:3400"]
```

## RPC Requirements

The tesseract relayer makes use of rpc methods that may not be exposed by default. For evm chains, ensure that the RPC API supports the following namespaces:

 - [`debug`](https://geth.ethereum.org/docs/interacting-with-geth/rpc/ns-debug)
 - [`eth`](https://geth.ethereum.org/docs/interacting-with-geth/rpc/ns-eth)

For substrate chains, ensure the RPC [enables unsafe rpc methods](https://docs.substrate.io/build/remote-procedure-calls/#safe-and-unsafe-rpc-methods).

The Hyperbridge node must additionally expose `offchain_localStorageGet`
(typically via `--rpc-methods Unsafe`). Without it the outbound task
cannot read accepted BEEFY consensus proofs and will log
`Method not found`.

## Withdrawing Fees

Tesseract maintains a local sqlite database where it keeps track of successfully delivered messages. The withdrawal process happens in two phases, the first phase is the fee accumulation on hyperbridge, then withdrawal on any of the connected chains.
In the fee accumulation phase, tesseract submits state proofs of messages all delivered to hyperbridge. If the proof verification is successful, the total amount of unclaimed fees for the relayer address is updated onchain. Next, the relayer can submit a withdrawal transaction to hyperbridge. This initiates potentially multiple ISMP requests to any chain where it has unclaimed fees, that instructs the Hyperbridge contract to credit the relayer their owed funds. This request will not timeout, allowing it to be submitted to the destination chain at any time.

### Accumulating fees

Fee accumulation runs automatically in the background. Every successful
outbound submit produces delivery receipts that flow into a per-chain
accumulation task; each task submits proofs of delivery to Hyperbridge
and increments the relayer's unclaimed-fee balance on-chain. Receipts
are persisted to the local database first so a crash between submit
and accumulation doesn't lose the reward.

> The legacy standalone `tesseract --accumulate-fees` subcommand no
> longer exists in the consolidated relayer — accumulation is part of
> the long-running process.

### Initiating withdrawals

To initiate a withdrawal from hyperbridge, a relayer needs to submit a transaction to hyperbridge which triggers the withdrawal request. This extrinsic is unsigned and will not require any native tokens for execution fees. Once the extrinsic is executed, hyperbridge dispatches a POST request that when executed on its destination, will provide the relayer with the fees they've accrued. The relayer account must have sufficient funds to deliver this request to its destination chain.

<br />
Run the `withdraw` subcommand to execute a single end-to-end withdrawal
pass and exit:

```bash lineNumbers
tesseract --config=$HOME/config.toml --db=$HOME/tesseract.db withdraw
```

`relayer.minimum_withdrawal_amount` still gates — chains whose
unclaimed balance is below the threshold are skipped.

### Automatic accumulation and withdrawals

The relayer also runs background tasks for automatic fee accumulation and withdrawals. Whenever a batch of messages is successfully delivered, the fee accumulation task receives the delivery receipts and starts the process of accumulating the fees on hyperbridge. This process happens concurrently for all successfully delivered message batches. For redundancy, the delivery receipts are stored in the database prior to accumulation so they can be retried manually if any error is encountered.

<br />
Withdrawing fees from hyperbridge is triggered at fixed intervals based on the
configured `withdrawal_frequency` and `minimum_withdrawal_amount`. Feel free to
the adjust these values as desired. The task will only make a withdrawal attempt
if your balance on hyperbridge is greater than or equal to the configured
`minimum_withdrawal_amount`. Any failed withdrawal attempts will be retried each
time the withdrawal task is triggered. The manual `withdraw` subcommand
described above can be used as a fallback when errors are encountered by the
automated task.

On each withdrawal tick, the relayer submits the withdrawal extrinsic
to Hyperbridge, waits for a `ProofAccepted` event at a height ≥ the
withdrawal block, reads the consensus proof from Hyperbridge's offchain
storage, and submits `[Consensus, Request]` as a single batch to the
destination — so it verifies the state proof and credits the relayer
in one transaction.
