# Hyper Fungible Token

HyperFungibleToken is a cross-chain token standard where each token deployment is its own bridge application. There is no shared custody pool, no governance overhead, and no token-governor — the token contract itself handles cross-chain messaging via [ISMP](/protocol/ismp).

## Architecture

The typical deployment uses a `WrappedHyperFungibleToken` on the token's home chain and a `HyperFungibleToken` on every remote chain. This keeps canonical supply on the home chain — the WrappedHFT accumulates locked tokens as users bridge out and releases them when users bridge back.

## Multi-Chain Topology

Using CREATE2 with the same salt and deployer across all chains gives each contract the same address, simplifying peer registration and frontend integration.

## Cross-Chain Message Format

Both contracts encode the same ABI message body, ensuring full interoperability between `HyperFungibleToken`, `WrappedHyperFungibleToken`, and [`pallet-hyper-fungible-token`](/developers/polkadot/hyper-fungible-token):

```solidity lineNumbers
struct Message {
    bytes from;     // original sender (for timeout refunds)
    bytes to;       // recipient on destination chain
    uint256 amount; // token amount
    bytes data;     // optional calldata for CallDispatcher
}
```

## CallDispatcher

Both contracts support optional calldata execution on the destination chain. If the `data` field in `SendParams` is non-empty, the contract calls `ICallDispatcher.dispatch(data)` after minting/unlocking tokens. This enables composable cross-chain interactions like transfer-and-swap or transfer-and-stake. Existing `CallDispatcher` deployments are listed on the [contract addresses](/developers/evm/contract-addresses/mainnet) page.

## ERC165 Interface Detection

Both contracts implement ERC165 for on-chain type detection:

| Contract | Interface ID |
|----------|-------------|
| `IHyperFungibleToken` | `0x7200c457` |
| `IWrappedHyperFungibleToken` | `0xe23d9765` |

The [TypeScript SDK](/developers/sdk/hyper-fungible-token) uses this to automatically determine which approval flow is needed.

## Security Model

### Source Verification

Every incoming message is verified against the registered peer:

```solidity lineNumbers
bytes memory expectedSource = _supportedChains[request.source];
if (keccak256(request.from) != keccak256(expectedSource)) revert UnauthorizedSource();
```

Only messages from registered peer contracts are accepted. An attacker cannot spoof messages from unregistered chains or impersonate the peer contract address on a registered chain — ISMP provides cryptographic proof that the message was actually dispatched on the source chain.

### Host Authorization

All ISMP callbacks (`onAccept`, `onPostRequestTimeout`) are protected by the `onlyHost` modifier inherited from [HyperApp](/developers/evm/api/hyperapp). Only the ISMP host contract can call these functions.

### Set-Once Host

The host address can only be set once via `configure()`. After the first call, subsequent calls cannot change the host — only the dispatcher can be updated.

## Pausability

Both contracts inherit from OpenZeppelin's `Pausable`. The owner can pause and unpause all operations:

```solidity lineNumbers
IHyperFungibleToken(tokenAddress).pause();
IHyperFungibleToken(tokenAddress).unpause();
```

When paused, `HyperFungibleToken` blocks all ERC20 transfers (`transfer` and `transferFrom`) in addition to cross-chain `send()` and `onAccept()`. This is a full stop on all token movement — use it for emergency situations or during migrations.
